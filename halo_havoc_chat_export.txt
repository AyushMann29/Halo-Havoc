### 🌌 Project Overview
**Halo Havoc** is a real-time multiplayer cooperative bullet hell game. 
* **Frontend**: Built using React, Vite, and **p5.js** (within `sketch.js`) to manage a high-performance rendering canvas and input handlers.
* **Backend**: Node.js, Express, and **Socket.io** (within `server.js`) running a centralized physics loop at 20 ticks per second (20Hz).

### 🛠️ Key Topics to Cover in an Interview
In a Full Stack Interview, developers look for optimizations, trade-offs, and architecture decisions. You can confidently discuss these three core engineering highlights implemented in the codebase:

1. **Object Pooling Optimization**
   * *What to say*: To support thousands of bullets and minions simultaneously without triggering the JavaScript Garbage Collector (GC)—which causes gameplay frame stutters—the backend uses pre-allocated arrays (pools) and toggles active/inactive flags instead of instantiating new objects.
2. **Delta Compression & Precision Truncation**
   * *What to say*: To optimize network bandwidth over WebSockets, the game state is packed into flat, ordered arrays rather than verbose JSON. It rounds coordinates to one decimal place to save characters, and applies delta compression so that player positions are only sent to clients if they actually changed.
3. **Collision Detection & Spatial Hashing Trade-offs**
   * *What to say*: The codebase implements spatial hashing to partition the game world into grid cells. However, in this specific game setup (max 8 players, 3000 bullets), spatial hashing actually incurs *more* string allocations (`'x,y'`) and lookups than simply iterating through the 8 active players. This shows you understand when *not* to over-engineer optimizations.

## 🤖 Assistant:
To understand **Object Pooling**, imagine you are running a busy restaurant:

### 🍽️ The Restaurant Analogy
* **Without Object Pooling (Bad Approach)**:
  Every time a customer orders food, you go to the store, buy a brand-new ceramic plate, serve the food, and when they finish eating, you **throw the plate in the trash**.
  * **The Problem**: Buying plates takes time and money (wastes CPU/memory), and your trash fills up so fast that you have to constantly pause the kitchen to take it out (this is the **Garbage Collector** running, which causes game stuttering/lag).

* **With Object Pooling (Good Approach)**:
  Before the restaurant even opens, you buy **50 plates** and put them in a cupboard. When a customer orders, you grab an empty plate from the cupboard. When they are finished, you wash it and put it back in the cupboard. 
  * **The Benefit**: You never buy new plates during dinner rush, and you never throw anything away. The cupboard is your **Object Pool**.

### 💻 How it works in your code
In your game, bullets are the "plates." 

1. **Creating the Cupboard**: When a new game room starts, the server initializes a fixed array of **3,000 inactive bullets** (`server.js`):
   ```javascript
   enemyBullets: Array.from({ length: 3000 }, () => ({
     active: false,
     x: 0,
     y: 0,
     ...
   }))
   ```
2. **Serving the Plate (Spawning a bullet)**: When the boss shoots, the server runs `spawnEnemyBullet`. Instead of using `new Bullet()`, it searches the pool for a bullet where `active` is `false`, changes its coordinates, and flips its status to `true`.
3. **Washing the Plate (Recycling)**: When a bullet goes off-screen or hits a player, the server doesn't delete it. It simply sets `bullet.active = false` so it can be reused for the next shot.

### 🌟 Why this matters for your interview
In JavaScript, creating and deleting thousands of objects per second forces the browser/server engine to run **Garbage Collection (GC)** to clean up memory. When the GC runs, it halts code execution for a few milliseconds, causing the game to stutter. Object pooling guarantees a **flat memory footprint** and smooth 60 FPS gameplay.

## 🤖 Assistant:
Here is a high-level summary of the hosting, resource usage, and capacity planning for your project, which are crucial for system design/architecture interview questions:

### 🌐 Hosting Strategy
* **Frontend (Vite/React)**: Deploy to a CDN like **Vercel**, **Netlify**, or **AWS S3 + CloudFront**. Because it's static, it is practically free, loads instantly, and handles infinite web traffic.
* **Backend (Node.js/Socket.io)**: Must be hosted on a **stateful virtual machine or container** (such as **AWS EC2**, a **DigitalOcean Droplet**, or **Render/Fly.io**). You **cannot** use serverless hosting (like AWS Lambda or Vercel functions) because WebSockets require a persistent, open TCP connection.

### 📊 Capacity Planning (On a single $5–$10/month VM: 1 vCPU, 2 GB RAM)
1. **Network Bandwidth limit ($\approx 1,000$ active players)**
   * Every room tick sends a packed array state update 20 times per second (20Hz).
   * Average payload is **~4 KB** per tick (highly optimized).
   * Bandwidth per player is ~80 KB/sec (~640 Kbps).
   * On a typical VM with a 1 Gbps network cap, the theoretical maximum is ~1,500 players, but in practice, you cap network utilization at 60–70% (~1,000 players).
2. **CPU Processing limit ($\approx 80–100$ active players) ── The Primary Bottleneck!**
   * Since Node.js is single-threaded, it can only run calculations sequentially.
   * Running a room's physics, boss scripts, and collision loops takes **1.5ms to 2.5ms** of CPU time per tick.
   * With a **50ms tick window**, a single thread can only process up to 20 active rooms in parallel before experiencing lag (2.5ms $\times$ 20 = 50ms).
   * At 4 players per room, a single-core VM is bottlenecked at **80–100 concurrent players**.
3. **RAM/Memory Limit (Not a bottleneck)**
   * Node.js baseline: ~30 MB.
   * Overhead per room (including the 3,000 bullet object pool allocation): ~1.5 MB.
   * 100 rooms (400 players) only consume ~180 MB of RAM, leaving plenty of head-room on a 2 GB VM.

### 📈 How to Scale to 10,000+ Players
If asked how you would scale this backend horizontally to handle massive traffic:
1. **Load Balancers with Sticky Sessions**: Run multiple server instances and route traffic using NGINX or AWS Application Load Balancer (ALB). You *must* configure **Sticky Sessions (Session Affinity)** so a player's socket connection stays routed to the same VM holding their active room.
2. **Redis Adapter**: Use a **Redis Pub/Sub adapter** (`socket.io-redis`) so different server instances can share game lobby lists and forward events between rooms across different VMs.
3. **Node.js Clustering**: Use Node's built-in `cluster` module to spawn one worker thread per CPU core, maximizing the multi-core utilization of your server virtual machines.
